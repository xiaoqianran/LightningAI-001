# 02 — 最小简体中文 MoE 训练计划

> 目标：在 Lightning AI 上，用**尽量少的钱与卡时**，训出一个**能跑通、能说话（简体中文）、结构上是 MoE** 的最小模型。  
> 原则：同卡型优先最便宜商家；能小就不大；先 smoke 再正式训。  
> **算力预算（用户确认）：** 可使用 **A100 / H100 / RTXP_6000**，但 **单价必须 &lt; $3 / 小时**；同卡型多家只选更便宜的。

---

## 1. 成功标准（Definition of Done）

| 级别 | 标准 | 说明 |
|------|------|------|
| **P0 必达** | 训练 Job 成功跑完 ≥1 个有效 epoch/step 区间 | loss 下降趋势可见 |
| **P0 必达** | checkpoint 可加载 | 本地或 Studio 能 `generate` |
| **P0 必达** | 简体中文可生成 | 对 5～10 条中文 prompt 有连贯简体输出（不要求高质量） |
| **P1** | 路由统计 | 各 expert 有负载记录（非死专家） |
| **P2** | 与 dense 小模型对比 | 同参数量级下 MoE 的 loss/速度随记 |

**非目标（本阶段不做）：**

- SOTA 中文能力、对齐/RLHF、多机万卡  
- 完整 MoE 论文复现（Switch-Transformer 全套）  
- 对外正式发布权重（可选后续）

---

## 2. 「最小 MoE」规格建议

面向 **教学 + 可训练**，推荐 **从零预训练一个玩具级 MoE**，而不是直接上 7B×8 专家。

### 2.1 推荐配置（v0 / 默认）

| 项 | 建议值 | 理由 |
|----|--------|------|
| 总参数（激活） | ~5M–30M 激活 / 总参可略大（有 &lt;$3/h 大卡时可略放大） | 单卡 A100/H100/RTX 6000 级绰绰有余 |
| 层数 | 4–8 Transformer blocks | 够学 token 共现 |
| 隐层 dim | 256–512 | 控显存 |
| Attention heads | 4–8 | — |
| **Experts 数 E** | **4 或 8** | 最小能体现 MoE |
| **Top-k** | **1 或 2** | top-1 更简单；top-2 更常见 |
| FFN 位置 | 每层或每隔一层 MoE-FFN | 间隔层可再省算力 |
| Vocab | 中文 BPE/SentencePiece **8k–32k** | 专为简体中文；勿直接套 GPT-2 英文词表 |
| 序列长度 | 128–512 | 先短后长 |
| 目标数据 | 简体中文纯文本 50MB–1GB | 先小后大 |

> **MoE 要点：** 同一套 attention；FFN 换成多个 expert；router 选 top-k；加 **load balancing loss**，避免所有 token 挤进一个 expert。

### 2.2 备选路线（若不想从零写架构）

| 路线 | 优点 | 缺点 | 何时用 |
|------|------|------|--------|
| **A. 自研最小 MoE（推荐）** | 全可控、Job 简单、最便宜 | 能力弱 | 学习 + 验证平台 |
| **B. 小开源 MoE 继续预训练** | 起点更高 | 权重大、卡要更好 | 有预算且要像样中文 |
| **C. 用 HF 小 dense + 改成 MoE FFN** | 组件现成 | 改动调试成本 | 熟悉 HF 时 |

**默认走路线 A。**

## 3. 数据计划（简体中文）

### 3.1 原则

- **只要简体**（可用规则/opencc 过滤繁体）  
- 先 **干净小集** 再扩量  
- 合法可用来源；不把私有数据默认进公开仓库  

### 3.2 分阶段数据

| 阶段 | 规模 | 用途 | 示例来源方向 |
|------|------|------|----------------|
| **D0 Smoke** | 1–5 万行 / ~10–50MB | 过拟合自检：loss 应明显下降 | 自建小语料、公开中文维基采样、教材级公开集 |
| **D1 Mini-train** | 100MB–1GB | 真正第一版 checkpoint | 过滤后的公开中文语料子集 |
| **D2（可选）** | 更大 | 略提流畅度 | 再扩；仍控在单卡可训 |

### 3.3 预处理流水线

1. 清洗：去 HTML、异常字符、过短/过长行  
2. 简繁：统一为简体  
3. 训练 **中文 tokenizer**（SentencePiece unigram/BPE）  
4. 打成 `train.bin` / `val.bin` 或 HF `datasets` arrow  
5. 固定 `seed`，划分 train/val（如 98/2）

**产物路径约定（Lightning）：**

```text
/teamspace/jobs/<job>/artifacts/
  tokenizer/
  checkpoints/
  logs/
  metrics.json
```

Studio Job 也可用 `/teamspace/jobs/<name>/artifacts`。

## 4. 训练方法

### 4.1 目标与损失

- 主损失：语言模型 **cross-entropy（next-token）**  
- 辅助：`λ * load_balancing_loss`（Switch Transformer 风格即可）  
- 优化器：AdamW；cosine 或恒定小 lr  
- 精度：优先 **fp16/bf16**（看卡是否支持 bf16）  

### 4.2 训练日程（逻辑阶段，不是一上来全量）

| 阶段 | 名字 | 算力 | 内容 | 通过标准 |
|------|------|------|------|----------|
| **S0** | 本地/CPU 单元测试 | CPU 或已有 hello 链路 | 前向、router、一步 backward | 无 NaN，grad 正常 |
| **S1** | GPU Smoke Job | **&lt;$3/h 池内最便宜**（优先 A100/H100/RTXP_6000 中达标者） | 极小模型 + D0 数据，几百～几千 step | loss 下降；存 ckpt |
| **S2** | Mini 正式训 | 同上，守 &lt;$3/h | v0 配置 + D1 数据，固定 step/时间盒 | val loss 有平台；中文生成可读 |
| **S3** | （可选）加长/加宽 | 仅当 S2 不够 | 略增 dim/数据；仍守单价上限 | 对比 S2 报告 |

### 4.3 时间与费用估算（公开数据 + 可断点续训）

**已确认：** 公开中文语料；**checkpoint resume 仍支持**；**默认 Job 不可中断（on-demand）**；GPU 单价 **&lt; $3/h**（A100/H100/RTXP_6000 优先池）。

下列为 **工程估算**（非报价单）。假设 v0 小 MoE（约 10M–30M 激活参数、seq 256–512、单卡、bf16/fp16）。小模型在 A100/H100 上往往 **算力吃不饱**，墙钟更多受 **数据加载、step 数、Job 冷启动** 影响，而不是「卡有多顶级」。

#### 吞吐粗算（量级）

| 机型档（&lt;$3/h 有货时） | 有效 token/s（小 MoE 粗量级） | 说明 |
|--------------------------|-------------------------------|------|
| T4 / L4 | ~5k–30k | 小卡也能很快跑完玩具模型 |
| A100 / RTXP_6000 | ~20k–100k+ | 常 bound 在 Python/数据而非 FLOPs |
| H100 | 同级或略高 | 对 v0 未必明显缩短墙钟 |

中文 SentencePiece：约 **1.5–3 字/token** 量级 →  
`50MB 纯文本 ≈ 1.5e7–3e7 tokens`；`500MB ≈ 1.5e8–3e8 tokens`。

#### 阶段墙钟（推荐默认）

| 阶段 | 数据 | 训练量（建议） | **纯训练** | **Job 总墙钟**（含拉起/装依赖/存盘） | **费用上限粗算**（@$3/h 封顶） |
|------|------|----------------|------------|--------------------------------------|--------------------------------|
| **S0** | 无 / 几条 | 1 个 step | &lt;1 min | 本地几分钟 | $0 |
| **S1 Smoke** | D0 **20–50MB** 公开子集 | 2k–5k steps 或 **1 个小 epoch** | **5–20 min** | **20–45 min** | **≈ $1–2.3**（45min×$3 上限粗算） |
| **S2 Mini** | D1 **200–500MB** 公开过滤集 | **1–2 epoch** 或 **20k–50k steps** | **30–90 min** | **1–2.5 h** | **≈ $3–7.5**（按实际单价×时长；封顶按 $3/h） |
| **S2+ 可选** | ~1GB | 2–3 epoch | **1.5–4 h** | **2–5 h** | **≈ $6–15** |

**更可能的「一次成功路径」中位估计：**

| 项目 | 估计 |
|------|------|
| S1 一次跑通 | **~30 min** 墙钟，**&lt; $1.5**（若实际 ~$1–2/h 则更低） |
| S2 出可用中文 ckpt | **~1.5–2 h** 墙钟，**约 $2–6**（取决于实时单价与是否抢占） |
| S1+S2 合计（顺利） | **~2–3 h** GPU 时间，**约 $3–8** 量级 |
| 含 1 次失败重跑 / Pending | 预留 **×1.5～2** 缓冲 → 建议心理预算 **$8–15** 整次实验 |

#### 断点续训如何影响时间

- **默认 on-demand（不可中断）**，避免 spot 排队/被踢。  
- ckpt 仍每 N step 保存，便于失败后 resume。  
- S2 `max_runtime` 建议 3h 硬封顶。

#### 时间盒（硬封顶，防跑飞）

| 阶段 | max_runtime 建议 | 说明 |
|------|------------------|------|
| S1 | **45–60 min** | 超时 stop；应已有 smoke ckpt |
| S2 | **2–3 h** | 不够则 resume 再提一个 Job 续跑 |
| 单 Job 费用警戒 | 时长 × 单价 **接近预算则 stop** | 脚本里 `stop_on_timeout=True` |

#### 为何不估「H100 一定更快很多」

v0 MoE 太小，H100 相对 A100/RTXP_6000 的加速比往往 **远小于价格比**。在都 &lt;$3/h 时，**选最便宜有货的**，总费用通常优于死磕 H100。

## 5. GPU / 商家选择策略（按你的偏好 + 预算）

### 5.0 硬约束（已确认）

| 约束 | 值 |
|------|-----|
| **允许的高端卡** | `A100`、`H100`、`RTXP_6000`（Lightning 名：RTX PRO 6000 档） |
| **单价上限** | **&lt; $3 / 小时**（超过则不用，即使更快） |
| **同卡多家** | 只选**更便宜**的商家/云，不选贵的 |
| **多卡** | `_X_2` 等若单价仍 &lt;$3/h 才考虑；默认仍单卡 |

提交前必须核对 **当时报价**（UI / pricing 页 / Job 创建页）。报价会变；以提交瞬间为准。  
若 A100/H100/RTXP_6000 当前均 ≥ $3/h → **降级**到单价 &lt;$3 的次优卡（L40S / L4 / T4 等）。**不使用可中断/spot。**

### 5.1 卡型选择流程

```text
候选池 = {A100, H100, RTXP_6000}  ∩  { 报价 < $3/h }  ∩  { 当前有货 }
若候选池非空:
  在池内选「$/h 最低」；同价再比：排队短 > 显存大 > 吞吐高
若候选池为空:
  扩大到其它 GPU（L40S/L4/T4…）中报价 < $3/h 且最便宜有货
  绝不提交 ≥ $3/h 的机型
```

| 阶段 | 策略 | 说明 |
|------|------|------|
| S0 | CPU | 单测 |
| S1 Smoke | 候选池内**最便宜**一档；无则任意 &lt;$3/h 小卡 | 目标是跑通，不炫卡 |
| S2 Mini | 候选池内最便宜；可略拉长墙钟 | 仍守 &lt;$3/h |
| 多卡 | 仅当单卡小时价×卡数逻辑清晰且总价策略仍省 | 默认不做 |

**不必为了「最小 MoE」强上 H100**——若 A100/RTXP_6000 明显更便宜且 &lt;$3，优先便宜的。

显存不够时顺序：**减 batch → 减 seq → 减 dim → 再在 &lt;$3/h 池内换更大显存卡**。

### 5.2 同卡型多家：一律偏便宜

```text
1. 固定 machine 类型（如 A100）
2. 过滤报价 ≥ $3/h 的商家
3. 剩余商家中选最便宜且有货
4. Pending 过久 → 同卡次便宜商家，或同预算内换机型
5. 不因「大厂」主动选贵的
```

可选省钱：

- **默认 on-demand**；勿默认 spot  

### 5.3 提交前口头确认模板

```text
机型: T4 | L4 | A100 | ...
报价: $X.XX / h  （必须 < 3.00）
商家/云: <同卡最便宜有货>
模式: on-demand（不可中断）
预计上限: <时长> / max_runtime  → 预计费用上限 ≈ 单价 × 时长
数据: D0|D1
配置: smoke|mini
```

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 全部 token 进同一 expert | load balance loss；监控 histogram |
| 中文乱码/繁简混 | 专用 tokenizer + 简体过滤 |
| NaN / 不收敛 | 降 lr、warmup、梯度裁剪、先 overfit 小集 |
| Job 启动慢 | 镜像预装依赖；数据不要塞进巨大 Studio 快照 |
| 烧钱 | 时间盒、小卡、便宜商家、max_runtime |
| 数据版权 | 只用明确可训公开集 / 自有数据 |
